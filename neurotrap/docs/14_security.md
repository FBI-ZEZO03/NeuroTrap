# Security Guide

This document covers the security controls built into NeuroTrap, host hardening recommendations, and operational security for running a public-facing honeypot.

---

## Network Isolation Model

The most critical security guarantee is network segmentation. A compromised honeypot must not be able to reach the analytics backend.

```
honeypot-net (172.20.0.0/24) ã external bridge
    Hosts: cowrie, opencanary, galah, nginx
    ã Can receive inbound attacker traffic
    ã Can NOT reach elk-net or management-net

elk-net ã internal bridge (no default gateway)
    Hosts: mongodb, behavior-engine
    ã No outbound route

management-net ã internal bridge (no default gateway)
    Hosts: api, nginx
    ã No outbound route

monitor-bridge (172.25.0.0/24) ã non-internal bridge
    Hosts: mongodb (static IP 172.25.0.10), packet-monitor (host network)
    ã Narrow bridge for host-network monitor to reach MongoDB only
```

Docker `internal: true` removes the container's default gateway. Containers on internal networks can communicate with each other but cannot initiate any outbound connection.

## Host Hardening

### SSH Relocation

Cowrie takes port 22. Your admin SSH must be on a different port:

```bash
# In /etc/ssh/sshd_config:
Port 50402
PermitRootLogin prohibit-password   # or 'no' for stricter
PasswordAuthentication no
PubkeyAuthentication yes
MaxAuthTries 3
AllowUsers <your-admin-user>
```

### UFW Firewall

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 50402/tcp   # admin SSH
sudo ufw allow 22/tcp      # Cowrie
sudo ufw allow 23/tcp      # Cowrie Telnet
sudo ufw allow 80,443,8080/tcp
# ... (full port list in installation guide)
sudo ufw enable
```

### Fail2Ban (Recommended)

Install Fail2Ban to protect the admin SSH port from brute-force:

```bash
sudo apt-get install fail2ban
```

```ini
# /etc/fail2ban/jail.local
[sshd]
enabled = true
port = 50402
maxretry = 4
bantime = 1h
findtime = 10m
```

> Do **not** configure Fail2Ban on ports 22/23 ã those belong to Cowrie and should accept all attempts.

### System Updates

```bash
sudo apt-get update && sudo apt-get upgrade -y
sudo apt-get install unattended-upgrades
sudo dpkg-reconfigure --priority=low unattended-upgrades
```

## Application Authentication

### JWT Tokens

The Flask API issues JWT tokens via `flask-jwt-extended`. Tokens are signed with `JWT_SECRET` from `.env`.

- Default expiry: 1 hour
- Stored in `localStorage` by the dashboard (not HttpOnly cookies ã tradeoff for SPA simplicity)
- Admin and analyst roles are encoded in the JWT claims

### Built-in Users

| Role | Default Username | Default Password | Access |
|------|-----------------|-----------------|--------|
| Admin | `admin` | `neurotrap2024` | Full ã all read + write + block |
| Analyst | `analyst` | `analyst2024` | Read-only ã all GET endpoints |

**Change defaults immediately:**
```bash
ADMIN_PASS=<strong-new-password>
ANALYST_PASS=<strong-new-password>
```

### MFA/TOTP

Optional TOTP second factor for admin logins:

```bash
# 1. Generate a secret
docker compose exec api python3 -c "import pyotp; print(pyotp.random_base32())"

# 2. Add to .env
MFA_ENABLED=1
MFA_SECRET=<base32-secret>

# 3. Get QR code for your authenticator app
curl -H "Authorization: Bearer $ADMIN_JWT" https://<host>/api/auth/otp/setup
# Returns base64 QR PNG ã display in any image viewer or authenticator

# 4. Restart API
docker compose restart api
```

When `MFA_ENABLED=1`, every admin login requires a TOTP code:
```json
{ "username": "admin", "password": "...", "otp": "123456" }
```

## SSL/TLS

Nginx terminates TLS. The default setup uses a self-signed certificate generated by `scripts/generate_ssl_cert.sh`.

**For production, replace with a CA-signed certificate:**

```bash
# Using certbot (Let's Encrypt)
sudo certbot certonly --standalone -d your-domain.com
cp /etc/letsencrypt/live/your-domain.com/fullchain.pem config/nginx/cert.pem
cp /etc/letsencrypt/live/your-domain.com/privkey.pem config/nginx/key.pem
docker compose restart nginx
```

## MongoDB Security

MongoDB is not exposed to the internet. It listens only on Docker internal networks.

Access requires authentication:
```bash
MONGO_USER=admin
MONGO_PASS=<strong-random-password>
MONGO_URI=mongodb://admin:<MONGO_PASS>@mongodb:27017/neurotrap?authSource=admin
```

Never expose port 27017 through UFW.

## Container Security

| Control | Configuration |
|---------|--------------|
| No privileged flag | All containers run without `--privileged` |
| NET_ADMIN/NET_RAW | Only `packet-monitor` needs these; scoped to that container |
| Docker socket | Only `deception-engine` mounts `/var/run/docker.sock` (to spawn decoy containers) |
| Read-only configs | All honeypot configs mounted `:ro` |
| No root | Cowrie runs as uid 999; other containers use non-root users where possible |

## Secrets Management

| Secret | Where It Lives | Notes |
|--------|---------------|-------|
| `MONGO_PASS` | `.env` (gitignored) | Use a strong random value |
| `JWT_SECRET` | `.env` (gitignored) | 64-char hex minimum |
| `SECRET_KEY` | `.env` (gitignored) | Flask session key |
| `ADMIN_PASS` | `.env` (gitignored) | Change from default |
| `ANTHROPIC_API_KEY` | `.env` (gitignored) | Never commit |

The `.gitignore` explicitly excludes `.env`, `data/`, and `*.pcap` files. Use `.env.example` as a template for new deployments.

## Attacker Data Privacy

Captured attacker sessions (commands, credentials, keystrokes) are stored in MongoDB. Consider:

- **Data retention:** Set TTL indexes on `alert_events` for high-volume collections (see `neurotrap-db/docs/08_retention_strategy.md`)
- **PCAP files:** Forensic captures land in `/tmp/forensic_*.pcap` ã consider moving to encrypted storage
- **Legal context:** In most jurisdictions, honeypot data is legally collectible. Consult local laws if operating in regulated environments

## Lynis Hardening Check

The performance target is a Lynis hardening score > 70:

```bash
sudo apt-get install lynis
sudo lynis audit system
```

Common improvements that raise the score:
- Enable AppArmor profiles
- Configure audit daemon (`auditd`)
- Enable process accounting
- Disable unused kernel modules
- Enable login banners

## Security Incident Response

If an attacker escapes a honeypot container:

1. **Immediate:** `docker compose down` to stop all containers
2. **Isolate:** Block the attacker IP at the UFW level (`ufw deny from <IP>`)
3. **Investigate:** Review `/var/log/syslog`, docker logs, and `alert_events` for pivot attempts
4. **Restore:** Rebuild from clean Docker images (`docker compose pull && docker compose up -d`)
5. **Post-incident:** Check if the escape route exploited a container misconfig (privileged flag, docker socket exposure, etc.)
